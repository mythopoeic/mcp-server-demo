# Bedrock (InvokeModel) + Haiku 4.5 behind a provider abstraction for the stretch tool

The `extract_orders` stretch tool is the only place the server itself calls an LLM. We route it through **Amazon Bedrock** using the classic `AnthropicBedrock` client — the `bedrock-runtime` **`InvokeModel`** API — with a cross-region **inference-profile** model id (`us.anthropic.claude-haiku-4-5-20251001-v1:0`, region required), behind a one-method provider interface with an **Anthropic-direct adapter as fallback**. We get schema-valid JSON via a **forced single tool call** — one tool whose `input_schema` *is* the requested JSON schema, with `tool_choice` pinned to it, reading the `tool_use` input.

**Why:** The OneMagnify JD lists AWS Bedrock as a hard requirement and rewards "practical tradeoffs between model providers and configurations" plus "output reliability." Bedrock-primary lets the demo truthfully claim "this calls Claude on Bedrock"; the Anthropic-direct fallback is demo insurance (one config flip if Bedrock hiccups). Haiku 4.5 is right-sized — extraction from an already-compressed grid is simple and well-bounded, and Haiku gives sub-second latency (important for a live demo) at ~5× lower cost than Opus; the model id is a config value so we can live-swap to the matching Opus inference profile to demonstrate the abstraction. Forced tool-use turns "I called an LLM" into "I built a reliable LLM service."

**InvokeModel vs. Mantle, and why it matters (live finding):** The first cut used `AnthropicBedrockMantle` (the `bedrock-mantle…api.aws` native-Messages endpoint). It worked — but the calls were **invisible to Bedrock model-invocation logging and CloudWatch metrics**, which only observe the classic `bedrock-runtime` `InvokeModel`/`Converse` APIs. For an enterprise demo, "you can't see it in CloudWatch" is disqualifying, so we switched to the `AnthropicBedrock` (InvokeModel) client. That path requires a cross-region **inference profile** for on-demand Haiku 4.5 — the bare foundation-model id 400s with "on-demand throughput isn't supported" — which is itself the correct enterprise setup (cross-region capacity + a single governable model handle). Verified: an InvokeModel call now produces a `bedrock-invocation-logs` record (`operation: InvokeModel`, the inference-profile ARN, input/output, identity) within seconds.

**Structured outputs vs. forced tool-use (live finding):** An earlier cut used the dedicated structured-outputs API (`output_config.format` + JSON schema). It passed the unit suite (fake provider) but a live call returned `400 — output_config.format: Extra inputs are not permitted`: the Bedrock endpoint does not accept `output_config` today. Forced tool-use is the portable equivalent — supported uniformly across Bedrock and Anthropic-direct and across model versions — and yields the same schema-valid JSON. This is exactly the kind of provider-specific gap the fake-provider seam can't catch and the eval harness exists to surface.

**Scale note (region-chunking):** The hero sheet holds ~426 orders; extracting them all in one call overruns the output-token budget (and the SDK refuses non-streaming calls that could exceed 10 min). `extract_orders` instead fans out **one bounded call per region** (the sheet's natural partition), runs them concurrently, and merges — recomputing `total_revenue` as the authoritative sum. Each region (~115 orders ≈ 10k output tokens) fits comfortably under a 20k `max_tokens` cap.

**Considered and rejected:** The `AnthropicBedrockMantle` endpoint (works, but its calls don't appear in model-invocation logging / CloudWatch — no observability). Hardwiring Anthropic-direct (misses the Bedrock requirement). Defaulting to Opus 4.8 (over-sized; signals failure to right-size cost/latency — the opposite of the JD's stated value). No abstraction (a Bedrock outage the night before would sink the demo's centerpiece with no fallback). One-shot extraction of the whole sheet (overruns the token budget; serial and slow; no parallelism win).
